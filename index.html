Data notes — 3 deviations from the task brief, and why

1. Currency is split, not summed. Amazon returns in this window span GBP, USD, EUR, CAD and a group with no currency recorded. The brief's Step 1 / Step 2 SQL sums refunded_amount with no currency grouping, which adds pounds to dollars to euros and labels the result £. That is the same error the brief's own scope table forbids, so every figure here is scoped to the single currency selected above. No total on this page mixes currencies, and no total mixes Amazon with eBay.

2. eBay SKUs are attributed on order_id + item_id, not order_id alone. order_transaction holds one row per line item, so the brief's Step 4 join duplicates a return once per SKU in the basket — 42 orders fan out, worst case one order across 10 line items, inflating the eBay SKU total by +12.7% (£12,825.91 against a true £11,384.44) and blaming innocent SKUs that merely shared a basket. Joining on item_id resolves 440 of 489 returns (90%) to exactly one SKU; the 46 that land on variation listings (one eBay listing, several SKUs) are split evenly, and 3 returns with no matching order line are shown as an explicit Unattributed row rather than dropped. The SKU tables now reconcile to the reason tables exactly, in every currency.

3. Reason values differ from the brief's list. The live data mostly carries a CR- prefix (CR-NOT_COMPATIBLE), and the same reason appears both with and without it — grouping raw would split one reason across two rows. The prefix is stripped mechanically so they merge. No reasons were renamed or combined beyond that: AMZ-PG-BAD-DESC is left as its own reason, though it is very likely the "not as described" signal the Listing team is looking for — worth confirming with DWC before acting on it.
